#01 AI, Machine Learning & Deep Learning — Clarifying the Stack
▾
💾 Auto-saved to browser

The Umbrella: Artificial Intelligence

Artificial Intelligence is the broad field of building systems that can reason, perceive, plan, and act in ways we associate with human intelligence. It is not a single algorithm or model — it is a research discipline spanning decades. Every time you interact with a recommendation engine, a spam filter, or a language model, you are using AI.

Machine Learning: Learning from Data

Machine Learning is a subset of AI. Instead of encoding every rule explicitly, ML systems discover patterns by being exposed to examples. You feed a model thousands of emails labelled "spam" and "not spam", and it learns to classify new emails on its own. The three core paradigms are:

  • Supervised Learning — labelled input-output pairs. The model learns a mapping Y = f(X). Examples: image classification, fraud detection, customer churn prediction.
  • Unsupervised Learning — no labels. The model discovers structure: clusters, anomalies, latent dimensions. Examples: customer segmentation, anomaly detection, topic modelling.
  • Reinforcement Learning — an agent takes actions in an environment and receives reward signals. It learns policies that maximise cumulative reward. Examples: game-playing (AlphaGo), robotics, recommendation ranking.

Deep Learning: Neural Networks at Scale

Deep Learning is a subset of Machine Learning that uses multi-layered neural networks. What makes it "deep" is the number of stacked transformation layers — each layer learns progressively more abstract representations of the data. Deep learning powers modern computer vision, speech recognition, and all large language models.

AI (broadest) └── Machine Learning ├── Classical ML (SVMs, Random Forests, Gradient Boosting) └── Deep Learning ├── CNNs (vision) ├── RNNs / LSTMs (sequential data, pre-2017) └── Transformers (language, vision, audio — 2017→present) └── Large Language Models (GPT-4, Claude, Gemini, LLaMA…)

Why the Distinction Matters in Practice

When a stakeholder says "let's add AI", they usually mean one of three very different things: (1) a classical ML model predicting a business metric, (2) a fine-tuned deep learning model for a perception task, or (3) an LLM for language generation. These have completely different cost profiles, infrastructure requirements, and failure modes. Knowing where on the stack a solution lives is the first step in making good architectural decisions.

#02 Narrow AI vs AGI — Where We Actually Are
▾
💾 Auto-saved to browser

Narrow AI (Artificial Narrow Intelligence)

Every AI system deployed in production today — including GPT-4, Claude 3.5, Gemini 1.5, and LLaMA 3 — is Narrow AI. These systems are extraordinarily capable within their training distribution but cannot autonomously generalise to domains they were not trained on. A language model that writes flawless code cannot drive a car. A vision model that identifies cancer in chest X-rays cannot negotiate a contract.

Generative AI vs Predictive AI

Reasoning Models — 2025 Development

A newer category has emerged: Reasoning Models (OpenAI o3, Claude 3.7 Sonnet with Extended Thinking, DeepSeek R1). These perform an internal chain-of-thought — often spending thousands of tokens "thinking" before producing an answer. They significantly outperform standard models on mathematics, code, and multi-step logical problems. The trade-off is latency and cost: a reasoning step might take 10–60 seconds. The practical rule: use reasoning models when correctness matters more than speed.

🧠 Self-Check Quiz 2 questions
Q1What is the key architectural difference between a classical ML model like a Random Forest and a Deep Learning model like a Transformer?
ANSWERRandom Forests combine many shallow decision trees, each making predictions on feature subsets, then vote. Transformers are multi-layer neural networks where each layer applies self-attention + feedforward transformations, learning hierarchical abstractions. DL models require far more data and compute but learn feature representations automatically — no manual feature engineering.
Q2Your team wants to predict which API integrations are likely to fail in the next 24 hours. Would you use Predictive or Generative AI? Justify your choice.
ANSWERPredictive AI. The task is structured binary/probabilistic classification on historical telemetry (latency, error rates, payload sizes). A gradient-boosted model or LSTM on time-series data is interpretable, fast, cheap, and well-suited. Generative AI would be expensive, non-deterministic, and architecturally unnecessary — you don't need language generation to produce a risk score.

#03 The Transformer Architecture — From 2017 to GPT-4
▾
💾 Auto-saved to browser

Origin: "Attention Is All You Need" (2017)

Before 2017, sequence modelling used Recurrent Neural Networks (RNNs) and LSTMs. These processed tokens one at a time, making GPU parallelism impossible and causing long-range dependency problems — by the time the model processed token 500, the signal from token 1 had largely vanished. The Google Brain paper "Attention Is All You Need" replaced recurrence entirely with a mechanism called self-attention, enabling full parallelism across the entire sequence.

Self-Attention: The Core Mechanism

Self-attention allows every token to directly attend to every other token in one parallel operation. Each token is mapped to three vectors:

  • Query (Q) — "what am I looking for?"
  • Key (K) — "what do I advertise about myself?"
  • Value (V) — "what information do I actually deliver?"
Attention(Q, K, V) = softmax( Q·K^T / sqrt(dk) ) · V

The sqrt(dk) term (square root of key dimension, typically 64) prevents dot products from becoming so large they push softmax into saturation — a region with near-zero gradients where learning stalls.

Multi-Head Attention

Instead of one attention operation, transformers run H attention heads in parallel. Each head projects Q, K, V into a different subspace and learns different relationship types simultaneously — one head might track subject-verb agreement, another long-range coreference, another syntactic structure. The outputs are concatenated and projected:

MultiHead(Q,K,V) = Concat(head_1, ..., head_h) · W_o

#04 Tokenisation & Context Windows
▾
💾 Auto-saved to browser

What Is a Token?

A token is the atomic unit of text that an LLM processes. Tokens are not words — they are sub-word fragments produced by the tokeniser. The most common algorithm is Byte Pair Encoding (BPE), which iteratively merges the most frequent character pairs in the training corpus until a target vocabulary size (~50K–100K tokens) is reached.

Tokenisation Examples
"Hello world"              →  ["Hello", " world"]               2 tokens
"integration"              →  ["integ", "ration"]               2 tokens  
"MuleSoft"                 →  ["Mul", "eSoft"]                  2 tokens
"API-led connectivity"     →  ["API", "-led", " connect", "ivity"] 4 tokens
"1000 tokens ≈ 750 words"  →  1 token ≈ 0.75 English words

Why Tokenisation Matters for Developers

  • API pricing is per token (input + output separately)
  • Context window limits are in tokens, not characters or words
  • Code tokens out more densely than English prose
  • Non-English languages typically use more tokens per word (2–4×)

Context Windows in 2025

Model Context Window Notes ───────────────────────────────────────────────────── GPT-4o 128K tokens OpenAI flagship Claude 3.5 Sonnet 200K tokens Anthropic flagship Claude 3 Opus 200K tokens Most capable reasoning Gemini 1.5 Pro 1M tokens Google DeepMind Gemini 1.5 Flash 1M tokens Faster, cheaper LLaMA 3.1 405B 128K tokens Meta open-source Mistral Large 128K tokens European open-source

The Long-Context Trap

Having a 1M token context window does not mean you should use all of it. LLMs suffer from a well-documented "lost in the middle" problem — they are better at recalling information placed at the start or end of a long context than information buried in the middle. For precise retrieval, structured RAG over chunked documents typically outperforms dumping everything into one giant prompt.

Prompt Caching (2024–2025)

Anthropic and OpenAI both offer prompt caching: if the first N tokens of a request are identical to a recent request, the KV-cache computation is reused, reducing latency and cost by up to 90% on the cached prefix. This is critical for applications with large static system prompts (e.g., a policy document loaded for every request).

Anthropic — Enabling Prompt Caching

      "cache_control": { "type": "ephemeral" }

🧠 Self-Check Quiz 2 questions
Q1GPT-4o has a 128K token context window. A developer wants to load a 400-page PDF (~120,000 words) into a single prompt. What are the technical and quality risks?
ANSWERTechnical: 120,000 words × (4/3 tokens/word) ≈ 160,000 tokens — exceeds the 128K window. Even if chunked to fit: (1) Lost-in-the-middle degradation: recall accuracy drops for content not near start/end. (2) Cost: at $5/1M input tokens, this single call costs ~$0.80 with no caching. (3) Latency: first-token latency increases with context length. Better approach: chunk the PDF and use RAG with semantic retrieval.
Q2Why can't you simply count words to estimate token cost for non-English content?
ANSWERBPE vocabulary is optimised for the training distribution, which is heavily English-dominant. Non-English scripts (Arabic, Chinese, Hindi) and technical jargon (chemical formulae, code in non-Latin scripts) map to fewer common subword merges, requiring more tokens per word — often 2–4×. Always use the provider's tokeniser (tiktoken for OpenAI, Anthropic's counter) to get accurate estimates before production deployment.

Sampling, Temperature & Hallucination

How LLMs generate text probabilistically — and why hallucination is an architectural property, not a bug.

⏱ ~30 min
#05 Temperature, Top-p and Decoding Strategies
▾
💾 Auto-saved to browser

How Text Generation Works

After the transformer processes all input tokens, the language model head projects the final hidden state into a vector of logits — one value per token in the vocabulary (50,000+ items). A softmax transforms these logits into probabilities. The model then samples the next token from this distribution. This is repeated token by token until a stop condition is met.

Temperature

Temperature is a scalar applied to the logits before softmax. It controls how flat or peaked the probability distribution is:

P_scaled(token) = softmax( logit / temperature )
Temperature Effect on Distribution Enterprise Use Case ────────────────────────────────────────────────────────── 0.0 Deterministic (greedy) Classification, JSON extraction 0.1 Near-deterministic Data transformation, field mapping 0.3 Focused Structured API responses 0.7 Balanced General assistants, Q&A 1.0 Creative Content generation, brainstorming 1.5+ Chaotic / incoherent Avoid in production

Top-p (Nucleus Sampling)

Top-p restricts sampling to the smallest set of tokens whose cumulative probability reaches threshold p. At p=0.9, if the top 3 tokens already account for 90% probability mass, no other tokens are considered. This prevents sampling extremely unlikely tokens while still allowing dynamic vocabulary size. Recommended: temperature 0.1–0.3 + top-p 0.9 for structured enterprise tasks.

Top-k Sampling

Restricts sampling to the k most probable tokens regardless of their cumulative probability. Less nuanced than top-p — a fixed k=50 behaves very differently when the model is confident (one token at 95%) vs. uncertain (50 tokens each at 2%). Top-p is generally preferred.

#06 Hallucination — Root Causes and Mitigations
▾
💾 Auto-saved to browser

What Is Hallucination?

Hallucination describes LLM output that is fluent and confident but factually wrong. A model might invent a citation, give an incorrect API method name, state a false date, or fabricate a company's revenue figure. Understanding why this happens is prerequisite to mitigating it.

Root Cause: Probabilistic Generation

LLMs are trained to predict the most statistically likely next token given prior context — not to retrieve verified facts. When a model doesn't "know" something precisely, it generates the most plausible-sounding completion. There is no internal mechanism that checks whether a statement is true before generating it.

Categories of Hallucination

Type Example Cause ──────────────────────────────────────────────────────────── Factual fabrication "The RFC was published in 2019" Wrong date Citation invention Cites non-existent paper Training pattern Reasoning error "2+2=5 because..." Chain of error Sycophancy Agrees with a false premise Alignment training Temporal error Uses outdated info Training cutoff Intrinsic Conflates two real entities Representation merge

Architectural Mitigations

  • Retrieval-Augmented Generation (RAG) — retrieve verified documents and inject them into the prompt. The model grounds its answer in provided evidence rather than recalled training weights. This is the most effective mitigation for factual grounding.
  • Tool Calling / Function Calling — instead of recalling a value, the model calls a tool (API, database query, calculator) that returns authoritative data. Never rely on the model to "remember" a current price, availability, or status.
  • Structured Output + Schema Validation — constrain output to a JSON schema. Validate fields programmatically before any downstream use. Reject non-conforming outputs and retry.
  • Citation Requirements — instruct the model to cite the exact passage it is drawing from. Uncited claims can be flagged for human review.
  • Low Temperature for Factual Tasks — temperature 0 makes generation deterministic; while it doesn't eliminate hallucination, it removes the sampling noise that amplifies uncertainty.
  • Self-Consistency Sampling — generate multiple independent answers at higher temperature, then take the majority answer. Expensive but effective for high-stakes decisions.
⚠ Critical — Hallucination is not a bug that will be patched in the next model version. It is an intrinsic property of autoregressive generation. Design every AI-powered workflow with the assumption that the model will sometimes be wrong, and build validation layers accordingly.
🧠 Self-Check Quiz 2 questions
Q1You are building an API that classifies incoming webhook payloads into one of five categories. What temperature setting do you use and why?
ANSWERTemperature 0.0 (or as close as the API allows). Classification requires deterministic, repeatable output — the same input must always produce the same category. Any temperature > 0 introduces sampling variance: the same payload might classify differently on different calls, breaking downstream routing logic. Use structured outputs / JSON schema to guarantee the response is always one of the five valid categories.
Q2An LLM correctly answers 9 out of 10 questions in testing. Your team argues this 90% accuracy means it's safe for production use in a workflow that writes records to a database. How do you respond?
ANSWER90% accuracy means 10% error rate. At 1000 requests/day that is 100 incorrect database writes per day — which may include fabricated IDs, wrong status codes, or invalid amounts. LLM integration requires: (1) output schema validation to catch structurally malformed responses, (2) business rule validation (amounts in range, IDs exist, status transitions valid), (3) HITL (human-in-the-loop) approval for high-impact write operations, and (4) audit logging for every AI-generated write. The 90% headline number is meaningless without understanding the failure modes of the 10%.

#07 Zero-Shot, Few-Shot & Chain-of-Thought
▾
💾 Auto-saved to browser

Zero-Shot Prompting

Provide only the task instruction with no examples. The model relies entirely on its pre-training to interpret what is being asked. Zero-shot works well for common tasks (summarisation, translation, basic classification) but degrades on domain-specific or multi-step tasks.

Zero-Shot Example
System: You are a JSON-only API response classifier.
User: Classify the following HTTP response as SUCCESS, CLIENT_ERROR, or SERVER_ERROR.
      Response: {"status": 404, "message": "Resource not found"}

Output: "CLIENT_ERROR"

Few-Shot Prompting

Provide 2–5 labelled examples before the target input. The model uses in-context learning to infer the pattern. Few-shot consistently outperforms zero-shot on nuanced tasks, especially when the output format is non-standard or the classification boundary is subtle.

Few-Shot Example
Classify the severity of each integration error.

Input: "Connection timeout after 30s to payment-service"
Output: {"severity": "HIGH", "requires_alert": true}

Input: "Duplicate message ID detected, processing skipped"
Output: {"severity": "LOW", "requires_alert": false}

Input: "TLS certificate expires in 3 days"
Output: {"severity": "MEDIUM", "requires_alert": true}

Input: "Circuit breaker open — downstream service unavailable"
Output: ???

Chain-of-Thought (CoT) Prompting

Instruct the model to work through reasoning steps explicitly before giving a final answer. Adding "Think step by step" or providing an example with explicit reasoning significantly improves accuracy on multi-step problems. Reasoning models (o3, Claude 3.7 Extended Thinking) perform this internally, but for standard models, explicit CoT instruction is essential for complex tasks.

Chain-of-Thought Instruction
System: When analysing an integration error, follow these steps:
  1. Identify the component that produced the error
  2. Determine whether it is a transient or permanent failure
  3. Check if the error is upstream (caller) or downstream (dependency)
  4. Recommend: Retry / Alert / Escalate / Log-only
  5. Output your final recommendation as JSON

User: Error: "Read timed out after 5000ms calling inventory-service GET /stock/SKU-8821"

Reasoning:
Step 1: HTTP client timeout in the integration flow
Step 2: Likely transient — network latency spike
Step 3: Downstream (inventory-service is a dependency)
Step 4: Retry with exponential backoff; alert if 3 consecutive failures

Output: {"action": "RETRY", "max_attempts": 3, "backoff_ms": 1000, "alert_threshold": 3}

Why Structured Output Matters for Integration

When LLMs are embedded in integration pipelines, the output must be machine-parseable. Prose responses that require downstream NLP to extract data create brittleness — any change in phrasing breaks the parser. The gold standard is constraining output to a JSON schema.

OpenAI Structured Outputs (2024)

OpenAI introduced guaranteed JSON schema compliance in August 2024. The model is constrained at the decoding level — it cannot produce tokens that would violate the schema.

🧠 Self-Check Quiz 2 questions
Q1You notice your zero-shot prompt produces inconsistent JSON — sometimes the model wraps the output in markdown code fences, sometimes it adds a prose explanation. What are three strategies to fix this?
ANSWER(1) Explicit format instruction: 'Output ONLY raw JSON. No markdown, no explanation, no preamble.' (2) Few-shot examples showing exact raw JSON output. (3) Use structured outputs / JSON mode (OpenAI) or tool-use pattern (Anthropic) which constrains generation at the decoder level — the model physically cannot produce non-JSON tokens. Option 3 is the most robust for production.
Q2A colleague says 'just add more instructions to the system prompt to prevent hallucination'. Why is this an incomplete solution?
ANSWERSystem prompt instructions are soft constraints — they influence but don't guarantee behaviour. LLMs can and do ignore instructions, especially under adversarial input (prompt injection) or when the instruction conflicts with strong training signal. Hard mitigations include: (1) output schema validation (programmatic, not LLM-enforced), (2) RAG grounding (model answers from retrieved evidence, not recalled weights), (3) tool calling (model fetches live data instead of recalling it). Instructions are a first line of defence, not a complete solution.

#09 Vector Embeddings & Semantic Search

What Is an Embedding?

An embedding is a dense numerical vector that encodes the semantic meaning of a piece of text. Text with similar meaning maps to nearby vectors in high-dimensional space, regardless of whether they share any words. This is what enables semantic search — finding documents by meaning rather than keyword overlap.

"The server failed to connect" │ Embedding model ▼ [0.82, -0.34, 0.71, 0.12, ..., -0.55] ← 1536-dimensional vector "Connection attempt to the host timed out" │ Embedding model ▼ [0.79, -0.31, 0.68, 0.15, ..., -0.51] ← very similar vector "The quarterly report is attached" │ Embedding model ▼ [0.11, 0.72, -0.23, -0.88, ..., 0.34] ← very different vector

Cosine Similarity

The standard metric for comparing embeddings is cosine similarity — it measures the angle between two vectors, normalised to remove magnitude effects:

cosine_similarity(A, B) = (A · B) / (‖A‖ × ‖B‖) → range: [-1, 1]

Similarity of 1.0 means identical direction (same meaning). 0 means orthogonal (unrelated). -1 means opposite (antonyms). In practice, semantic similarity above 0.85 indicates highly related content.

Keyword Search vs Semantic Search

The Problem RAG Solves

LLMs have a training cutoff date and no knowledge of your private documents, internal APIs, proprietary data, or events after training. RAG solves this by retrieving relevant context at query time and injecting it into the prompt. The model then answers from the retrieved evidence rather than from its parametric memory.

Hybrid Search

Combining BM25 keyword search with vector semantic search via Reciprocal Rank Fusion (RRF) consistently outperforms either approach alone — especially for short queries and queries with exact-match requirements like product codes, error codes, or API method names. Most production RAG systems use hybrid search.

↗ Original RAG Paper (Lewis et al., 2020)↗ Pinecone RAG Learning Hub↗ LangChain RAG Concepts
🧠 Self-Check Quiz 2 questions
Q1Your RAG system retrieves the right documents but the final LLM answer is still wrong. What are three places in the pipeline to investigate?
ANSWER(1) Chunk quality — chunks may be too small (missing surrounding context needed for the answer) or split mid-sentence. Try parent-document retrieval: retrieve small chunks but inject their parent passage into the prompt. (2) Re-ranking — the raw vector similarity may not surface the most answer-relevant chunk as #1. Add a cross-encoder re-ranker to re-order top-K by relevance to the specific question. (3) Prompt template — the model may not be explicitly instructed to answer ONLY from the provided context, leading it to blend retrieved evidence with parametric memory. Add: 'Answer ONLY using the provided context. If the answer is not in the context, say so.'
Q2Explain hybrid search and describe a specific scenario where pure vector search would fail but hybrid search would succeed.
ANSWERHybrid search combines BM25 (term frequency keyword search) with vector semantic search. Results are merged using Reciprocal Rank Fusion. Scenario: A user queries 'MULE-12345 error code'. Vector search looks for semantic similarity — it may return documents about error handling in general. BM25 finds exact matches for the string 'MULE-12345'. Pure vector search fails because the query is not semantic — it's a specific identifier. Hybrid search succeeds because BM25 anchors on the exact code while vector search catches related error documentation.

#11 What Is an AI Agent?

Definition

An AI agent is a system where an LLM is given a goal and autonomously decides what actions to take to achieve it — calling tools, reading results, updating its plan, and continuing until the goal is met or a limit is reached. The key distinction from a simple LLM call: the model drives a multi-step decision loop, not a single input-output pass.

The ReAct Loop (Reason + Act)

Most agent frameworks implement a cycle of: Think → Act → Observe → repeat.

GOAL
"Find the current price of EUR/USD and convert €500"
▼
THINK I need the current EUR/USD exchange rate
ACT call_tool("get_exchange_rate", {"pair":"EUR/USD"})
OBSERVE {"rate": 1.0842, "timestamp": "2025-03-05..."}
▼
THINK Now calculate 500 × 1.0842
ACT call_tool("calculate", {"expression":"500 * 1.0842"})
OBSERVE {"result": 542.10}
▼
OUTPUT ✓
"€500 converts to $542.10 USD at the current rate of 1.0842"

Tool Calling (Function Calling)

Instead of generating a prose response, the LLM returns a structured JSON object specifying which tool to call and with what arguments. The application executes the tool, sends the result back, and the LLM continues. This separates reasoning (LLM) from execution (application), which is the correct security boundary.

Tool Schema Definition
{
  "name": "get_api_health_status",
  "description": "Retrieves the current health status of a named API endpoint.
                  Returns: status (UP/DOWN/DEGRADED), latency_ms, last_error.",
  "parameters": {
    "type": "object",
    "properties": {
      "api_name": {
        "type": "string",
        "description": "The registered name of the API (e.g., 'inventory-api')"
      },
      "include_history": {
        "type": "boolean",
        "description": "Include last 5 status changes in response",
        "default": false
      }
    },
    "required": ["api_name"]
  }
}

Q1What is the critical security difference between a tool call being 'suggested by the LLM' and 'executed by the application'?
ANSWERThe LLM suggests a tool call via structured JSON output. The application is responsible for: (1) validating the tool name exists and is permitted for this user, (2) validating all arguments (type checking, range checking, enum validation), (3) applying RBAC — does this user have permission to call this tool with these arguments?, (4) executing the tool, (5) handling errors. The LLM is an untrusted component — it may hallucinate tool names, inject malicious arguments, or be manipulated via prompt injection in retrieved content. Never auto-execute tool calls without validation.

RAG implementation in MuleSoft uses two separate flows: an indexing flow that processes documents into a vector store, and a query flow that retrieves context and generates answers at runtime.

Phase 1: Indexing Documents

Mule XML — Document Indexing Flow
<flow name="rag-index-documents-flow">

  <!-- Trigger: scheduled job or admin API endpoint -->
  <http:listener config-ref="HTTP_Listener_config"
                 path="/admin/rag/index" allowedMethods="POST"/>

  <!-- Load documents from a source (file, DB, API) -->

             outputMimeType="text/plain"/>

    config-ref="MuleSoftAIChain_Config"

          level="INFO"/>

</flow>

<flow name="rag-query-flow">

  <http:listener config-ref="HTTP_Listener_config"
                 path="/ai/ask" allowedMethods="POST"/>

    config-ref="MuleSoftAIChain_Config"

    storageType="IN_MEMORY"

  <ee:transform>
    <ee:message>

          sources: payload.sources default [],

Q1AI generated a DataWeave transformation and it works on your test input. Before deploying to production, what DataWeave-specific checks should you run?
ANSWER(1) Null safety — test with inputs where optional fields are missing or null. AI often generates r.profile.first ++ " " ++ r.profile.last which throws on null. Generating MUnit Tests with AI

Writing MUnit tests is time-consuming — setting up mock connectors, defining test data, asserting on transformed payloads. AI can generate a complete test suite from a flow description or from the flow XML itself.

Prompt to Generate MUnit Test
Generate an MUnit 2.x test suite for this Mule flow.
The flow receives a POST request with a JSON order payload,
validates that the total is positive, calls an HTTP API to
check inventory, and returns 201 if successful or 409 if
inventory is insufficient.

Write test cases for:
1. Happy path (inventory available)
2. Inventory unavailable (expect 409)
3. Invalid payload — missing required fields (expect 400)
4. Downstream API timeout (expect 503)
AI-Generated MUnit Test (excerpt)
<munit:test name="test-happy-path-inventory-available"
             description="Order created successfully when inventory is available">

  <munit:behavior>
    <!-- Mock the inventory API call -->
    <munit-tools:mock-when doc:name="Mock Inventory Check"
      processor="http:request" doc:id="inventory-check">
      <munit-tools:with-attributes>
        <munit-tools:with-attribute attributeName="doc:id"
          whereValue="inventory-check"/>
      </munit-tools:with-attributes>
      <munit-tools:then-return>
        <munit-tools:payload value='{"available": true, "stock": 50}'
          mediaType="application/json"/>
        <munit-tools:attributes value='{"statusCode": 200}'/>
      </munit-tools:then-return>
    </munit-tools:mock-when>
  </munit:behavior>

  <munit:execution>
    <munit-tools:http-requester path="/orders" method="POST">
      <munit-tools:body value='{ "productId": "SKU-001", "quantity": 2 }'
        mediaType="application/json"/>
    </munit-tools:http-requester>
  </munit:execution>

  <munit:validation>
    <munit-tools:assert-that
      expression="#[attributes.statusCode]"
      is="#[MunitTools::equalTo(201)]"/>
  </munit:validation>

</munit:test>

🧠 Self-Check Quiz 1 questions
Q1Your team is considering using AI to generate 100% of your MUnit tests automatically. What is the fundamental risk with this approach and how would you mitigate it?
ANSWERThe fundamental risk is that the AI generates tests that match the implementation, not tests that validate the business requirements. If the flow has a bug, AI-generated tests written from the flow code will pass — they test what the code does, not what it should do. Mitigation: (1) Write tests from requirements documents / Gherkin scenarios, not from flow code. (2) Use AI to generate test scaffolding and mock setup, but have engineers write assertions from business rules. (3) Include negative tests and edge cases that a developer might not think to add. (4) Require test coverage of all documented API error responses, not just the happy path.

🧠 Self-Check Quiz 2 questions
Q1A senior developer is resistant to using AI assistance, arguing 'it will make junior developers lazy and produce code they don't understand'. How do you respond?
ANSWERThe concern is valid but the solution isn't to avoid AI — it's to use it thoughtfully. AI assistance at its best works like a senior pair programmer: it writes scaffolding and boilerplate while the developer makes design decisions, reviews output critically, and ensures correctness. The risk of developers accepting AI output without understanding is a training and process problem, not an inherent AI problem. Mitigations: (1) Require developers to explain AI-generated code in code reviews — they must understand it to defend it. (2) AI-generated DataWeave must pass MUnit tests written from requirements, not from the code. (3) Treat AI output as a draft requiring review, not final code. The developer who can critically evaluate and refine AI output is more valuable than one who refuses to use it.
Q2Design a prompt strategy for having an AI assistant review a MuleSoft flow for security and performance issues.
ANSWERProvide: (1) The full Mule flow XML, (2) The DataWeave transformations, (3) Connector configurations (with sensitive values redacted). Prompt: 'Review this MuleSoft flow for: SECURITY: (a) Are credentials referenced from secure properties or hardcoded? (b) Is PII in the payload logged? (c) Is the HTTP connector configured with TLS? (d) Are error responses leaking internal system details? PERFORMANCE: (a) Are there unnecessary synchronous HTTP calls that could be made async? (b) Are large payloads being processed in-memory that could be streamed? (c) Is caching used for repeated expensive calls? (d) Are there N+1 call patterns (loop + HTTP call)? Report each finding with: severity (HIGH/MEDIUM/LOW), location (flow name, line), and recommended fix.'
